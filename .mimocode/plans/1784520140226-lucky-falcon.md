# Migration Plan: Firebase → Payload CMS + Neon PostgreSQL

## Overview
Migrate the entire backend from Firebase (Firestore, Auth, Storage, Cloud Functions) to Payload CMS running on Neon PostgreSQL. This is a **complete backend replacement** — the frontend (Next.js pages, components, styling) stays but its data layer changes.

---

## Architecture Decision: Payload as Next.js Plugin

Payload CMS runs **inside** the Next.js app as a plugin (not a separate server). This means:
- Payload admin UI replaces the custom `/admin` pages
- Payload's Local API replaces all Firestore/Server Action calls
- Payload's Auth replaces Firebase Auth
- Payload's Upload plugin replaces Firebase Storage
- No separate backend server needed

**Target stack:**
```
Next.js 15 + Payload CMS 3.x + Neon PostgreSQL + Drizzle adapter
```

## Phase 2: Define Payload Collections (Data Schema)

Map all 22 Firestore collections → Payload collections:

### Master Data (simple, no relations)
| Firestore | Payload Collection | Key Fields |
|---|---|---|
| `languages` | `Languages` | name, code |
| `nationalities` | `Nationalities` | name, code |
| `provinces` | `Provinces` | name, country |
| `tourTypes` | `TourTypes` | title, description, icon, order |
| `guides` | `Guides` | name, phone, email, relation→languages, provinces, nationalities |

### Users & Auth
| Firestore | Payload Collection | Key Fields |
|---|---|---|
| `users` | `Users` (built-in) | email, password, role, displayName, status, avatar (upload) |

### Content (CMS core)
| Firestore | Payload Collection | Key Fields |
|---|---|---|
| `posts` | `Posts` | title, slug, content (richText), status, author→Users, categories→Categories, tags→Tags, seo, featuredImage→Media, scheduledFor |
| `categories` | `Categories` | name, slug, description, parent→self, order, count |
| `tags` | `Tags` | name, slug, description, count |
| `comments` | `Comments` | post→Posts, authorName, content, status, parent→self |
| `media` | `Media` (Payload uploads) | file, title, altText, caption, mediaType |

### Tourism
| Firestore | Payload Collection | Key Fields |
|---|---|---|
| `tours` | `Tours` | code, name, summary, itinerary, dates, guide→Guides, status, provinces, photos (upload array) |
| `feedback` | `Feedback` | name, country, rating, message, tour→Tours, status, source, guide→Guides |
| `reviews` | `Reviews` | authorDisplay, rating, message, tour→Tours, status, reviewType |

### Site Configuration
| Firestore | Payload Collection | Key Fields |
|---|---|---|
| `siteSettings` | `SiteSettings` (global) | siteName, hero fields, about, contact, social, colors, languages |
| `themeSettings` | `ThemeSettings` (global) | fonts, colors, header/footer styles, custom CSS/JS |
| `siteContentSlides` | `Slides` | locale, title, subtitle, imageUrl (upload), order, status |
| `navigationMenus` | `NavigationMenus` | key, locale, published, items (array of nested fields) |
| `stories` | `Stories` | title, excerpt, coverImage (upload), publishedAt, tags |

### Subcollections → Flatten or Nest
| Firestore Subcollection | Payload Approach |
|---|---|
| `posts/{id}/versions` | Payload built-in `versions` (admin-level autosave) |
| `posts/{id}/revisions` | Payload built-in `versions` (block-level revisions) |
| `navigationMenus/{id}/items` | Array field on NavigationMenus collection |
| `tours/{id}/comments` | Separate `TourComments` collection with relation→Tours |

### Cloud Functions → Payload Hooks/Lifecycle
| Cloud Function | Payload Replacement |
|---|---|
| `feedbackSubmit` | `afterOperation` hook on Feedback collection |
| `feedbackUploadComplete` | Upload lifecycle hook |
| `adminFeedbackApprove` | Custom endpoint or CollectionOperation hook |
| `adminFeedbackReject` | Custom endpoint or CollectionOperation hook |
| `postVersioning` | Payload built-in versions |
| `scheduledPublishing` | Payload jobs plugin (cron) |
| `emailNotifications` | Payload afterOperation hooks or custom API |
| `updateTaxonomyCounts` | afterChange hook on Posts to update category/tag counts |

## Phase 3: Auth Migration

### 3.1 Replace Firebase Auth → Payload Auth
- Payload has built-in authentication with JWT
- Users log in via Payload's `/admin` panel
- Custom login page: `src/app/(payload)/admin/login/page.tsx`
- Session management via Payload's JWT tokens

### 3.2 Replace `useFirebase` / `useUser` hooks
- Remove `src/firebase/provider.tsx` context
- Replace with Payload's `useAuth` hook from `@payloadcms/next`
- Update `AdminRouteGuard` to check Payload auth session

### 3.3 Protect admin routes
- Payload handles admin auth automatically
- Custom API routes use `payload.auth({ headers })` to verify

## Phase 4: Replace Data Fetching

### 4.1 Server-side (replaces Firebase Admin SDK)
**Before:** `initializeFirebaseAdmin()` → `admin.firestore().collection().get()`

Files to update:
- `src/lib/content-service.ts` → use Payload Local API
- `src/lib/actions.ts` → use Payload Local API
- `src/lib/navigation.ts` → use Payload Local API
- `src/lib/cms-actions.ts` → use Payload Local API
- `src/lib/finished-tour-comments.ts` → use Payload Local API

- `src/lib/search-utils.ts` → use Payload search or PostgreSQL full-text
- `src/app/api/finished-tours/[tourId]/comments/route.ts` → Payload custom endpoint

## Phase 7: Custom Endpoints & Business Logic

Replace Cloud Functions with Payload custom endpoints or hooks:

### 7.1 Custom endpoints (`src/app/(payload)/api/`)
```
/feedback/submit       → POST endpoint for public feedback form
/feedback/approve/:id  → POST endpoint for admin approval
/feedback/reject/:id   → POST endpoint for admin rejection
/finished-tours/:id/comments → GET/POST for tour comments
```

### 7.2 Collection hooks
```
Feedback afterCreate   → AI summary generation (keep Genkit)
Feedback afterUpdate   → Status change notifications
Posts afterChange      → Update category/tag counts
Posts afterChange      → Auto-publish scheduled posts (cron job)
Comments afterCreate   → Email notifications
```

### 7.3 Scheduled jobs (replaces PubSub triggers)
```
publishScheduledPosts  → Payload jobs plugin, runs every 5 min
cleanupTrashedPosts    → Payload jobs plugin, runs daily
```

## Implementation Order

1. **Phase 1** — Payload setup, config, admin panel working
2. **Phase 2** — All collection schemas defined
3. **Phase 3** — Auth working via Payload
4. **Phase 5** — Uploads working via Payload

6. **Phase 4** — Migrate data fetching (biggest phase, do collection by collection)
7. **Phase 6** — Remove Firebase completely
8. **Phase 8** — Clean up and verify

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Large codebase, many files to touch | Migrate incrementally: Payload runs alongside Firebase until fully replaced |
| Real-time subscriptions lost | Payload supports live queries via SSE; or use SWR/React Query polling |
| Cloud Function triggers need careful replacement | Test each hook individually |
| Data migration may have edge cases | Keep Firestore export JSONs as backup |
| Admin UI will look different | Payload admin is customizable, but different from current UI |

## Alternative: If Payload is Too Heavy

If the full Payload admin panel migration feels too large, a lighter approach:
1. Keep the custom admin UI as-is
2. Replace only the backend: use `@payloadcms/db-postgres` directly as a PostgreSQL client
3. Use Payload's Local API without the admin panel

This preserves the existing admin UI while only changing the database layer.
